mysql锁介绍

什么是锁

  • 对共享资源进行并发访问
  • 提供数据的完整性和一致性

每个数据库的锁实现完全不同

  • MyISAM表锁
  • InnoDB行锁
  • MSSQL行级锁 with 锁升级

latch

  • mutex
  • rw-lock

锁的区别

lock latch
对象 事务 线程
保护 数据库内容 内存数据结构
持续时间 整个事务过程 临界资源
模式 行锁，表锁，意向锁 读写锁，互斥锁
死锁 通过waits for graph，time out等机制进行死锁检测与处理 无死锁检测与处理机制，仅通过应用程序加锁的顺序(latch leveling)保证无死锁的情况发生
存在于 lock manager的哈希表中 每个数据结构对象中

latch是针对程序内部资源(比如，全局变量)锁的定义，而这里的lock针对的是数据库的事务

lock是有latch来保证和实现的

mysql锁的类型

S行级共享锁，X行级排它锁，IS意向共享锁，IX意向排它锁，AI自增锁

意向锁介绍

  • 揭示下一层请求的锁类型
  • IS：事务想要获得一张表中某几行的共享锁
  • IX：事务想要获得一张表中某几行的排他锁
  • innodb存储引擎中意向锁都是表锁

  1. 假设有事务tx1，需要对记录A进行加X锁：

    • 在该记录所在数据库上加一把意向锁IX
    • 在记录所在的表上加一把意向锁IX
    • 在记录所在的页上加一把意向锁IX
    • 最后在该记录A上加一把X锁

  2. 假设此时有事务tx2需要对记录B(假设和记录A在同一个页中)加S锁

    • 在该记录所在数据库上加一把意向锁IS
    • 在该记录所在表上加一把意向锁IS
    • 在该记录所在的页上加一把意向锁IS
    • 最后在该记录B上加一把S锁

  3. 假设此时有事务tx3需要在记录A上加S锁

    • 在该记录所在的数据库上加一把意向锁IS
    • 在该记录所在的表上加一把意向锁IS
    • 在该记录所在的页上加一把意向锁IS
    • 发现该记录被tx1事务锁定，那么tx3需要等待，直到tx1进行commit

  4. 总结：

    • 加锁是从上往下，一层一层加锁的
    • 意向锁都是相互兼容的，因为意向锁表示的是下一层在请求什么类型的锁

  5. 锁兼容

    锁兼容 IX X IS S
    IX 兼容 冲突 兼容 冲突
    X 冲突 冲突 冲突 冲突
    IS 兼容 冲突 兼容 兼容
    S 冲突 冲突 兼容 兼容

意向锁的作用

  • 意向锁是为了实现多粒度的锁，表示在数据库中不但可以实现行级别的锁，还可以实现页级别的锁，以及数据库级别的锁
  • 如果没有意向锁的话，当你去锁一张表的时候，需要对表下单所有记录都进行加锁操作，且对其他事务刚刚插入进来的记录(游标已经扫过的范围)就没发在上面加锁了，此时就没有实现锁表的功能
  • 上述锁的操作都是在内存中完成的，不会存放在数据库中，且大部分加的都是意向锁，都是兼容的，锁释放操作则是从记录开始从下往上进行释放的
  • innodb没有数据库级别的锁，页没有页级别的锁(innodb只能在表和记录上加锁)，所以innodb的意向锁只能加在表上，即innodb存储引擎中意向锁都是表锁

锁与并发

locking(锁)，concurrency control(并发控制)，isolation(隔离级别)，serializability(序列化)

以上四个概念其实在数据库中讲的就是同一个概念：锁是用来实现并发控制，并发控制是用来实现隔离级别，隔离级别是通过锁来实现的，锁的目的是为了使事务之间执行是序列化的

事务隔离级别

隔离级别 隔离级别的值 导致的问题
Read-Uncommitted 0 导致脏读
Read-Committed 1 避免了脏读，允许不可重复读和幻读
Repeatable-Read 2 避免了脏读。不可重复读，允许幻读
Serializable 3 串行化读，事务只能一个一个执行，避免了脏读，不可重复读，幻读。执行效率慢，使用时慎重。

  • 脏读： 一个事务对数据进行了增删改，但未提交，另一个事务可以读取到未被提交的数据，如果第一个事务这时候回滚了，那么第二个事务就读到了脏数据。

  • 不可重复读：一个事务中发生了两次读操作，第一个读操作和第二次读操作之间，另外一个事务对数据进行了修改，这个两次读取的数据是不一致的。

  • 幻读：第一个事务对一定范围的数据进行批量修改，第二个事务在这个范围增加了一条数据，这个时候第一个事务就会丢失对新增数据的修改。

  1. 不可重复读诗事务并发带来的问题，可重复读是事务隔离级别
  2. 隔离级别越高，越能保证数据的完整性和一致性，但是对并发性能的影响也越大。大多数的数据库默认隔离级别是Read-Committed，比如SqlServer, Oracle。少数数据库默认隔离级别为Repeatable-Read。比如 Mysql InnoDB
  3. 隔离级别越低，事务请求的锁越少，或者保持锁的时间就越短
  4. 隔离性：一个事务所在的修改，对其他事务是不可见的，好似是串行执行的
  1. 终端3在commit之前，查询不大终端2中的数据，在终端2中也查询不到终端3中的数据
  2. 终端3在commit之后，查询不大终端2中的数据，而终端2中可以查询到终端3中的数据
  3. 其实是不符合隔离性要求的，查询数据库的事务隔离级别select @@transaction_isolation
  4. 一般线上环境使用Read-Commited
