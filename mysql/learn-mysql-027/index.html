mysql锁介绍

什么是锁

  • 对共享资源进行并发访问
  • 提供数据的完整性和一致性

每个数据库的锁实现完全不同

  • MyISAM表锁
  • InnoDB行锁
  • MSSQL行级锁 with 锁升级

latch

  • mutex
  • rw-lock

锁的区别

lock latch
对象 事务 线程
保护 数据库内容 内存数据结构
持续时间 整个事务过程 临界资源
模式 行锁，表锁，意向锁 读写锁，互斥锁
死锁 通过waits for graph，time out等机制进行死锁检测与处理 无死锁检测与处理机制，仅通过应用程序加锁的顺序(latch leveling)保证无死锁的情况发生
存在于 lock manager的哈希表中 每个数据结构对象中

latch是针对程序内部资源(比如，全局变量)锁的定义，而这里的lock针对的是数据库的事务

lock是有latch来保证和实现的

mysql查看锁信息

mysql锁的类型

S行级共享锁，X行级排它锁，IS意向共享锁，IX意向排它锁，AI自增锁

意向锁介绍

  • 揭示下一层请求的锁类型
  • IS：事务想要获得一张表中某几行的共享锁
  • IX：事务想要获得一张表中某几行的排他锁
  • innodb存储引擎中意向锁都是表锁

  1. 假设有事务tx1，需要对记录A进行加X锁：

    • 在该记录所在数据库上加一把意向锁IX
    • 在记录所在的表上加一把意向锁IX
    • 在记录所在的页上加一把意向锁IX
    • 最后在该记录A上加一把X锁

  2. 假设此时有事务tx2需要对记录B(假设和记录A在同一个页中)加S锁

    • 在该记录所在数据库上加一把意向锁IS
    • 在该记录所在表上加一把意向锁IS
    • 在该记录所在的页上加一把意向锁IS
    • 最后在该记录B上加一把S锁

  3. 假设此时有事务tx3需要在记录A上加S锁

    • 在该记录所在的数据库上加一把意向锁IS
    • 在该记录所在的表上加一把意向锁IS
    • 在该记录所在的页上加一把意向锁IS
    • 发现该记录被tx1事务锁定，那么tx3需要等待，直到tx1进行commit

  4. 总结：

    • 加锁是从上往下，一层一层加锁的
    • 意向锁都是相互兼容的，因为意向锁表示的是下一层在请求什么类型的锁

  5. 锁兼容

    锁兼容 IX X IS S
    IX 兼容 冲突 兼容 冲突
    X 冲突 冲突 冲突 冲突
    IS 兼容 冲突 兼容 兼容
    S 冲突 冲突 兼容 兼容

意向锁的作用

  • 意向锁是为了实现多粒度的锁，表示在数据库中不但可以实现行级别的锁，还可以实现页级别的锁，以及数据库级别的锁
  • 如果没有意向锁的话，当你去锁一张表的时候，需要对表下单所有记录都进行加锁操作，且对其他事务刚刚插入进来的记录(游标已经扫过的范围)就没发在上面加锁了，此时就没有实现锁表的功能
  • 上述锁的操作都是在内存中完成的，不会存放在数据库中，且大部分加的都是意向锁，都是兼容的，锁释放操作则是从记录开始从下往上进行释放的
  • innodb没有数据库级别的锁，页没有页级别的锁(innodb只能在表和记录上加锁)，所以innodb的意向锁只能加在表上，即innodb存储引擎中意向锁都是表锁

锁与并发

locking(锁)，concurrency control(并发控制)，isolation(隔离级别)，serializability(序列化)

以上四个概念其实在数据库中讲的就是同一个概念：锁是用来实现并发控制，并发控制是用来实现隔离级别，隔离级别是通过锁来实现的，锁的目的是为了使事务之间执行是序列化的

事务隔离级别

隔离级别 隔离级别的值 导致的问题
Read-Uncommitted 0 导致脏读
Read-Committed 1 避免了脏读，允许不可重复读和幻读
Repeatable-Read 2 避免了脏读。不可重复读，允许幻读
Serializable 3 串行化读，事务只能一个一个执行，避免了脏读，不可重复读，幻读。执行效率慢，使用时慎重。

  • 脏读： 一个事务对数据进行了增删改，但未提交，另一个事务可以读取到未被提交的数据，如果第一个事务这时候回滚了，那么第二个事务就读到了脏数据。

  • 不可重复读：一个事务中发生了两次读操作，第一个读操作和第二次读操作之间，另外一个事务对数据进行了修改，这个两次读取的数据是不一致的。

  • 幻读：第一个事务对一定范围的数据进行批量修改，第二个事务在这个范围增加了一条数据，这个时候第一个事务就会丢失对新增数据的修改。

  1. 不可重复读诗事务并发带来的问题，可重复读是事务隔离级别
  2. 隔离级别越高，越能保证数据的完整性和一致性，但是对并发性能的影响也越大。大多数的数据库默认隔离级别是Read-Committed，比如SqlServer, Oracle。少数数据库默认隔离级别为Repeatable-Read。比如 Mysql InnoDB
  3. 隔离级别越低，事务请求的锁越少，或者保持锁的时间就越短
  4. 隔离性：一个事务所在的修改，对其他事务是不可见的，好似是串行执行的
  1. 终端3在commit之前，查询不大终端2中的数据，在终端2中也查询不到终端3中的数据
  2. 终端3在commit之后，查询不大终端2中的数据，而终端2中可以查询到终端3中的数据
  3. 其实是不符合隔离性要求的，查询数据库的事务隔离级别select @@transaction_isolation
  4. 一般线上环境使用Read-Commited

mysql中验证脏读

终端2和终端3中数据库的隔离级别设置read-uncommitted;

  • 终端2中开启事务，插入数据，未提交，终端3中可以查询到
  • 终端3中开启事务，插入数据，未提交，终端2中也可以查看到，说明出现了脏读
  • 终端2中事务回滚，终端3中查询不到终端2插入的数据了

mysql中验证不可重复读

终端2和终端3中数据库的隔离级别设置read-committed;

  • 终端2中开启事务，插入数据，未提交，终端3中查询不到数据
  • 终端2中事务提交，终端3中能查询到数据，说明RC解决了脏读问题
  • 终端2中在开启事务，修改数据，事务提交，终端3中能查询到修改后的数据，说明出现了不可重复读的问题

在rc隔离界别下，一个事务只要提交或者回滚了，另一个事务是可见的，没有提交，另一个事务中是查询不到的，也就是做解决脏读问题

mysql中验证幻读

终端2和终端3中数据库的隔离级别设置read-committed;

  • 终端2中开启事务，插入数据，提交，终端3在事务中是可以查询到的
  • 终端2中在开启事务，插入数据，提交，终端3在事务中可以查询到的，说明出现了幻读问题

mysql中验证repeatable read隔离解决姐幻读问题

终端2和终端3中数据库的隔离级别设置repeatable-read;

  • 终端2中开启事务，插入数据，未提交，终端3中开启事务，查询不到数据
  • 终端2中提交事务，终端3中页查询不到数据，如果终端3中结束事务，在开始事务，就能查询到数据，解决了脏读和不可重复读到问题
  • 终端2中在开启事务，插入新数据，未提交，终端3中开启事务，查询不到数据
  • 终端2中提交事务，终端3中页查询不到

rr隔离级别，一个事务虽然提交了，另一个事务如果没有提交，也是查询不到的，只有另一个事务提交后，在开启新事务，才能查询到，有savepoint概念

可以看出innodb到rr隔离级别，实现了事务的隔离性，也就是一个事务的操作对另一个事务不影响，完全隔离

  • 不可重复读：是在一个事务中，针对同一条记录，执行两次相同的sql，得到的结果不一致，针对的sql是update
  • 幻读：是在一个事务中，执行两次相同的sql，得到了不同的结果，针对是insert和delete

innodb中锁的算法

  • 每个事务每个页 一个锁对象，大小约100字节
  • 通过位图存放锁信息，内存占用少
  • 没有锁升级

record lock

  • 单个记录上的锁

  • 假设有记录10，30，50，且在30记录上加锁

    对于在记录30上加锁，表示30该记录上加了行锁

    [30,30]: 只锁定记录30自身

gap lock

  • 锁定的是范围，但不包括记录本身

  • 假设有记录10，30，50，且在30记录上加锁

    对于在记录30上加锁，表示在记录10到记录30之间加锁(10是30的前一个记录)

    (10, 10): 锁定该范围，不包含两个边界(不能插入15，20等这个范围内的数据，但是可以对10和30记录进行操作)

    gap lock解决了幻读问题

next-key lock

  • record lock + gap lock, 锁定一个范围，并且锁定记录本身

  • 假设有记录10，30，50，且在30记录上加锁

    对于在记录30上加锁，表示在记录10到记录30范围内加锁(gap lock)的同时，在记录30上同时也加上了行锁

    (10, 30]: 锁定该范围，且包含记录30本身(不能插入15，20等这个范围内的数据，同时对记录30不能进行操作，当记录10时可以删除或操作的)

锁重用

只有一把锁，不会出现两把锁

隐式锁

尽管终端2还没有事务还没有提交(没有创建内存对象)，但是可以通过trx_id来判断该记录上是否有锁

通过在线事务列表中是否存在该trx_id进行判断，因为事务没有提交(即活跃)，就表示在这个记录上是有锁的

注意：锁住的是索引

锁与隔离级别

repeatable read

  • lock mode IX: 表上有一个IX意向锁
  • lock_mode X：表示的是next key lock
  • 记录10上面有record lock
  • 记录11上面有record lock
  • 记录13上面有record lock
  • 记录20上面有record lock

即在RR隔离级别下，锁住的还是(负无穷大～10], (10, 11], (11, 13], (13, 20]

锁住20是因为在RR级别下，(<=13)将从表(B+树)的第一个记录(10)开始比对，一直比对到第一个出现大于13的记录(20)为止

    如果update这条sql被hang住了，就说明记录20被锁住了

总结：

  1. RR级别下的锁，默认是next key lock
  2. rr会对游标打开过的所有记录加锁

read committed

  • lock mode X locks rec but not gap: 表示的就是Record lock记录锁
  • 记录10上面有record lock
  • 记录11上面有record lock
  • 记录13上面有record lock

即在RC隔离级别下，锁住的是10，11，13记录本身(lock mode X locks rec but gap)， 会出现幻读问题

在大部分情况下，RC隔离级别下没有gap锁，但是一些场景下，可能会出现，设置了RC隔离级别后，binlog_format = row，否则会出现主从不一致的情况

next-key lock优化为record lock

  • 锁定是一条记录
  • 当索引含有唯一约束时

a是主键，具有唯一约束，且返回的记录只有一条

lock mode X locks rec bug not gap：表示的就是记录锁

只对13这个记录加了锁

  • next key lock可以降级为record lock是因为返回的记录具有唯一性，不存在幻读问题
  • 当唯一索引是复合索引时，且查询条件只包含部分列的话，其实还是有gap lock的

lock_mode X：由于b=2时复合索引的一部分，所以没办法降级为record lock，还是next key lock

非唯一索引列的等值查询

RR隔离级别

lock_mode X: next key lock

  • 记录10上面加了锁
  • 记录11上面加了锁
  • 记录13上面加了锁
  • 记录20上面加了锁

tbl_test_not_unique_rr_lock中存在系统自己生成的主键，当查询条件为a=13是，游标需要访问所有的记录(索引)，来判断主键对应a列是否等于13，即需要扫描所有的主键，则会将所有记录(索引)都锁住，形成了表锁的效果

在RR的隔离级别下，当查询的列没有索引时，会锁住所有记录

RC隔离级别

lock_mode X locks rec but not gap: 记录锁，锁的是index GEN_CLUST_INDEX

当查询条件时非索引列的等值查询，RR隔离级别下会锁住每条记录(形成表锁的效果)，而RC隔离级别下只锁住查询条件记录本身

假如RR没有Gap lock

如果在RR级别下，且没有gap锁，则此时tx1锁住的是1，3，5，7(记录锁)，并标记删除，同时tx2可以插入记录6，此时表中记录为6

而log走记录的是insert 6；del<=7; (先提交的在前面)，如果此时有slave机器进行同步，该表中记录为空，此时，主从机器上的数据不一致

当有了gap lock后，tx2的insert 6；需要等待tx1的del<=7;执行完成后才能执行，此时的log为del<=7; insert 6;也就不会出现数据不一致的问题

二级索引列的等值查询

RR隔离级别

index b 二级索引：next key lock

  • 二级索引 key = 3，value = 5主键

index PRIMARY 聚集索引：record lock

lock_mode X locks gap before rec： gap lock

即二级索引锁住的范围是(1,3], (3, 6), 主键索引只锁住了a=5这条记录[5,5]

如果不锁住(3,6)还是可以有别的事务添加b=3的记录，从而导致了幻读

  • 以下sql语句在终端3中的执行效果

    二级索引上加了record lock或next key lock，则对应的主键加 record lock

    二级索引加了gap lock，则对应的主键不加锁

    主键查询加next key lock，而二级索引查询加gap lock

RC隔离级别

lock_mode X locks rec but not gap: 记录锁

当查询条件时二级索引列的等值查询时

  • RC模式下，二级索引查询的记录上有一个记录锁，对应的聚集索引上也哟一个记录锁
  • RR模式下，二级索引查询的记录上有一个next key lock，该记录的下一个记录上有一个gap lock(二级索引上)，对应的聚集索引上有一个记录锁

RR模式下，二级索引上多了一个gap lock主要是为了防止插入二级索引列的值，产生幻读

插入意向锁(insert intention lock)

  • 插入意向锁本质上就是个gap lock
    • 普通gap lock不允许在(上一条记录到本记录)范围内插入数据
    • 插入意向锁gap lock允许在(上一条记录到本记录)范围内插入数据
  • 插入意向锁的作用是为了提高并发插入的性能，多个事务同时写入不同数据至统一索引范围(区间)内，并不需要等待是他事务完成，不会发生锁等待

插入的过程

假设现在有记录10，30， 50， 70；且为主键，需要插入记录25

  • 找到小于等于25的记录，这里是10

    • 如果记录中存在25，且带有唯一性约束，则需要在记录25上加S gap lock

    • 如果没有gap lock的话，就会存在插入两个25，破坏了唯一性约束

  • 找到记录10的下一条记录，这里是30

  • 判断下一条记录30上是否有锁

    1. 如果30上面没有锁，则可以插入
    2. 如果30上面有record lock，则可以插入
    3. 如果30上面有gap lock或next key lock，则无法插入，因为锁的范围是(10,30)/(10,30]; 在30上面增加insert intention lock (此时处于waiting状态)，当gap lock/next key lock释放时，等待的事务将被唤醒，此时记录30上才能获得insert intention lock，然后在插入25

    注意：一个事务insert 25且没有提交，另一个事务delete 25时，记录25上会有record lock

一致性的非锁定读

隔离级别为RU和SR时时不需要mvcc的，因此只有RC和RR时，才存在mvcc，在隔离级别为RC，RR时，读取(select)操作是不会加锁的，通过行多版本控制(mvcc)的方式读取当前执行时间点的记录，mvcc是指，innodb使用基于时间点的快照来获取查询结果，读取是在访问表上不设置任何锁。

一致性非锁定读在两种隔离级别RC和RR下，有什么不同，这两种隔离级别下，拍得快照时间点不同：

  • RC时，同一个事务内的每一个一致性读总是设置和读取自己的最新快照。也就是说，每次读取时，都重新拍得一个最新的快照，所以RC时总能读取到最新提交的数据。
  • RR时，同一个事务内的所有一致性读总是读取同一个快照，此快照是执行该事务的第一个一致性读时拍得的。

通过undo指针指向，可以读取前一个版本甚至前几个版本的记录(即通过undo来构造版本记录)，从而实现快照读(snapshot read)

通过trx_id判断该记录是否被锁住(在线事务列表)，从而决定是否要读取之前的版本(undo)

一致性锁定读

innodb提供两种锁定读，即select...for update和select...lock in share mode。它们两个都能提供额外的安全性。

select...for update：对于读取的行记录加一个X排他锁，其他事务不能对锁定的行加任何锁

select...lock in share mode：对于读取的行记录添加一个S共享锁。其它事务可以向被锁定的行加S锁，但是不允许加X锁，否则会被阻塞

这两种锁定读在搜索时所遇到的(注意不是最终结果集中的)每一条索引记录上设置排他锁和共享锁。此外，如果当前隔离级别时RR，它还会在每个索引记录前面的间隙上设置排他或共享的gap lock(排他和共享的gap lock没有任何区别，二者时等价的)

一致性非锁定读适用于对于数据一致性要求不是很高的情况，而一致性锁定读适用于对数据一致性要求比较高的情况，这个时候就需要对读操作进行加锁以保证数据逻辑一致性。

AI自增锁

  • 一个表一个自增列
  • auto_increment pk
  • 在事务提交之前就释放
    • 其他的锁是在事务提交时才释放
    • 如果AI锁在提交后才释放，那么其他事务就没发插入了，无法实现并发

AI自增锁示例

  1. 终端2在rollback之前，在终端3中执行insert语句，可以执行成功，说明AI锁已经释放了
  2. 在终端2的commit之后，执行sql查询，可以看到主键值时3，说明AI锁其实是提交了，导致自增需要是从3开始的
  3. 事务回滚后，自增值不会跟着回滚，导致自增值不连续。

AI自增锁的方式

插入类型 说明
insert-like insert-like指所有的插入语句，如insert，replace，insert...select，replace...select，load data等
simple inserts simple inserts指能在插入前就确定插入行数的语句。这些语句包括insert，replace等，需要注意的是，simple inserts不包含insert...on duplicate key update这类的sql语句
bulk inserts bulk inserts指在插入前不能确定插入行数的语句，如insert...select，replace...select，load data
mixed-mode inserts mixed-mode inserts指插入中有一部分的值是自增长的，有一部分是确定的，如果insert int t1 (c1, c2) values (1,'a'), (null,'b'),(5,'c'),(null,'d');也可以是insert...on duplicate key update类的语句
  1. 所有插入都是insert-like
  2. 如果插入能确定行数的，就是simple-inserts
  3. 如果插入前不能确定行数的，就是bulk-inserts
  4. 如果部分自增长，部分指定的，就是mixed-mode inserts
  5. insert...on duplicate key update不推荐使用，因为是非标准的sql，而且效果并非像预期的那样
设置自增的并发读

  • 参数--innodb_autoinc_lock_mode={0|1|2}

    innodb_autoinc_lock_mode是read-only的，修改后需要重启mysql实例

    • 0是传统方式

      1. 在sql语句执行完之后，AI锁才释放
      2. 可以保证自增值是连续的，sql执行完毕之前，其他事务是不能插入的

    • 1是连续(默认)

      1. buik inserts，通传统方式一样，在sql执行完毕之后，AI锁才释放
      2. simple inserts，并发方式，因为sql执行完之前，确定了自增值之后，就可以释放自增锁了

      因为bulk inserts不知道要插入多少行，所以只能等待insert结束后，才知道N的值，然后一次性(ai+N)

      而simpel inserts知道插入的行数(M)，所以可以先(ai + M)，然后将锁释放，给别的事务使用，然后在慢慢插入数据

    • 2是交错

      1. 所有自增都可以并发
      2. 同一个sql语句自增可能不连续

      • 这样做的好处是，对于批量的，耗时的插入，sql不会长时间持有AI自增锁的，而是插入一条(有且仅插入一条，而simple inserts是确定好M条)语句后就释放，这样可以给别的事务使用，实现并发

      • 但是这种方式并发读虽然增加了，但是性能不一定好，尤其是单线程导入数据时，因为要不断的申请和释放锁

      • 对于批量插入来说，自增就可能变得不连续了

自增列与索引

自增列必须定义为索引列，且必须是第一列，这样实现的目的是，因为索引的第一列是排序的，这样重启后，数据库就可以直接找到最后一个自增值，然后+1操作后，就可以作为下一个自增列的值了，不然的话，需要全表扫描，innodb的自增列的值不是持久化的

自增的两个参数

  • 步长 --auto_increment_increment = 1
  • 初识值 --auto_increment_offset = 1

两个服务器上的user表的主键值没有交叉了

如果有三台服务这，A:[offset=1, increment=3], B:[offset=2, increment=3], C:[offset=3, increment=3]

如果一开始并不知道后面有多少台服务器，这一开始可以把increment的值设置的大一些，比如10，这样只会浪费一些自增值

这样做的目的是保证每个节点上产生的自增值是全局唯一的，这样做并不是用来做双主的(比如一些额外的唯一索引能保证全局唯一么?)

SERIALIZABLE和REPEATABLE READ的区别

SERIALIZABLE是通过两阶段来保证隔离性的

  • 加锁阶段：只加锁，不放锁
  • 解锁阶段：只放锁，不加锁

且无论读还是写，都要加锁，但是这样做了以后，失去了mvcc的特性(一致性非锁定读)
