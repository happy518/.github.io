事务的ACID

  • A(Atomicity)原子性
    • redo
    • 事务内的sql要么都执行完毕，要么都全部回滚
  • C(Consistency)一致性
    • undo
    • 事务的开启和结束，没有破坏数据的结构和约束
  • I(Isolation)隔离性
    • lock
    • 一个事务的修改，对其他事务是不可见的，好似串行执行
  • D(Durable)持久性
    • redo & undo
    • 以及提交的事务是持久的

事务的类型

  • flat transaction(平事务)
    • 大部分都是这种类型的事务
  • flat transaction with savepoint(带保存点事务)
    • 事务内执行完部分操作后，可以save work，如果回滚的话，可以回滚到该save work到状态
    • 在mysql内非常有用
  • chained transaction(链式事务)
    • 在提交一个事务的时候，释放不需要的数据对象，将必要的处理上下文隐式的传给下一个要开启的事务
    • 提交事务操作和开始下一个事务操作合并成一个原子操作，则意味着下一个事务将看到上一个事务的结果，就好像在一个事务中执行的一样
    • 相关于在一个事务commit之后，立即打开了一个begin(系统自动)
  • nested transaction(嵌套事务)
    • innodb不支持
    • BDB支持
  • distributed transaction(分布式事务)
    • 在分布式环境下运行的扁平事务
    • 参与分布式的节点都要支持ACID
  • long transaction

事务演示

  • flat transaction

在存储过程中使用事务，需要使用start transaction，因为begin是存储过程中表示的是代码块开始的关键字

默认情况下，autocommit为on

  • flat transaction with savepoint

回滚到s1的保存点，且该事务没有rollback或者commit， 当前事务是没有结束的，只有执行commit或者rollback的时候，事务才结束

如果最后执行rollback，则整个事务rollback

如果最后执行commit，则s1之前的sql会commit，因为s1到s2之间到sql以及rollback了

事务以begin或者start transaction开始，以rollback或者commit结束

redo

redo就是常说的重做日志，用来实现持久性的

redo的组成

  • redo log buffer

    • --innodb_log_buffer_size，由log block组成，每个log block512字节(不需要double write)

  • redo log file

    • --innodb_log_file_size 单个innodb redo文件的大小，(推荐8G，官方建议等于buffer_pool_size)
    • 之前不建议调大的原因是因为有bug，如果调整太大，恢复速度很慢
    • 5.5版本的redo文件的总大小(num * size)是有限制的(4G)
    • 5.6以后调整为512G
    • 调大以后，唯一的内容就是恢复的内容变多了
    • 5.6以后，正常关闭mysql，然后增加或者缩小该值后，mysql自动扩展该文件大小
    • --innodb_log_files_group: 文件的数量
    • --innodb_log_group_home_dir: redo文件和数据文件分开，可以选择更快的磁盘

需要注意：size * in_group < 512G

redo文件的存放地址，默认和数据文件安在一起，线上建议分开

redo log buffer的刷新条件

  • master thread每秒刷新redo的buffer到logfile
    • 5.6版本以后，增加innodb_flush_log_at_timeout参数，可以设置刷新间隔，默认1
    • innodb_flush_log_at_timeout该值设置的越大，相对性能好一点(IO操作变少了)，但是万一发生宕机，丢失的数据就越多
  • redo log buffer使用量大于1/2时进行刷新

  1. 当ib_logfile0写满时，去写ib_logfile1

  2. 当ib_logfile1也写满了，会覆盖ib_logfile0

这样做的优点：

不需要归档，减少了IO操作

缺点：

如果redo_log_file大小，则可能需要等待，因为等待，因为要覆盖log_file中的log_block时，如果该log_block中的脏页还没有进行刷新的话，则需要等待这个脏页刷新完。

所以需要把redo log file设置的尽可能的大

如何判断innodb_log_file_size设置小了

Log sequence number: 当前内存中的lsn

Pages flushed up: 最后刷新到磁盘页上的最新lsn

两者之差就表示redo log还有多少没有刷新到磁盘

如果该差值接近重做日志的大小的75%时，表明你的innodb_log_file_size设置小了(75%左右就强制刷新了)

  • 事务提交时进行刷新

    • innodb_flush_log_at_trx_commit = {0|1|2}
    • 0是事务提交的时候并不打日志(redo log buffer)写入到磁盘，(1s或者大于1/2时刷新日志)
    • 1是事务每次提交的时候要确保日志(redo log buffer)写入磁盘，即使宕机，也可以通过redo恢复，达到持久性要求
    • 2是事务提交的时候，仅将日志(redo log buffer)写入到操作系统缓存

    1可以保证数据不丢失，0可能会丢失1s到数据，2如果是mysql停止，不会丢数据，当如果是系统宕机了，在缓存里的数据就丢失了，不建议设置2，建议设置为1

组提交

  • 一次fsync刷新多个事务

    • 性能提高10到100倍
    • 通过sysbench工具，测试update_non_index.lua(mysql5.5有bug，性能较差)

  • innodb存储引擎原生支持

    1. fsync

      • O_DIRECT仅对写数据时有用，redo日志是不会通过O_DIRECT方式写入到磁盘的，而是写到文件系统的缓存中
      • O_DIRECT仅仅写数据到磁盘，但是数据的元信息没有同步，fsync可以将元信息同步到磁盘
      • fsync可以将redo日志(redo log buffer)从文件系统的缓存同步到磁盘

      假设HDD磁盘的IOPS为100(即每秒只能执行100次fsync)，且一个事务中只有一条insert into的sql时，那么1秒就只能插入100条数据，即TPS只有100

      IOPS --> 决定 --> fsync --> 决定 --> TPS

      假设使用组提交，一次fsync可以刷新5个事务，那么在IOPS为100的情况下，也可以提交500个事务，这样性能就得到了提升(一次IO提交多个事务)

redo日志内容

redo日志时物理逻辑日志，根据页进行记录，记录的内容时逻辑的(页的变化)

rec body中记录时页的变化，并非sql语句

binlog

在my.cnf中的一个配置

为什么mysql中需要搞两份日志

  1. binlog在mysql成产生
  2. redo在innodb成产生

如果mysql只有一个innodb，其实只有redo日志是可以的，但是mysql还有其他的存储引擎，如果不使用binlog，意味着每个存储引擎都要实现一次binlog所提高的功能。

一次事务提交，既要写binlog，又要写redo log

查看当前binlog状态

File: 当前binlog写入的文件

Position：写入的偏移量，如果有新的数据插入，这个值就会改变(字节数)

binlog会不断的增大，超过设置的大小后(最后一个事务要执行完，可能大于设置的阈值)，就会分割binlog(一个事务不能跨过两个binlog)

binlog的Position表示当前该binlog的写入字节数

binlog的类型

  • statement
    • 易于理解
    • 记录sql语句(那如果有一些不确定的sql语句，类似uuid()函数，limit without order by，主从会出现不一致)
  • row
    • 设置RC隔离级别，必须选这个
    • 记录sql语句操作的哪些行(行的变化)
    • 每张表一定要有主键(性能较高)
    • 数据一致性高，可flashback
  • mixed
    • 混合statement和row格式，不推荐

binlog events

查看当前binlog中的内容

第一行是mysql的版本和日志版本

第二行是gtid号

上面两行记录在每个binlog文件的开头都有(mysql5.6以后)

binlog演示

  • statement

出现了query类型，并且能显示sql语句

  • row

Table_map:把表名映射到ID，并且记录列的类型和表的元数据信息

Write_rows：插入的类型，但是不能看到插入的数据

update操作对于的是Update_rows

delete操作，对应的是Delete_rows的操作

  • binlog_rows_query_log_events

    打开盖参数，可以在row格式下，看到对应的sql信息

多了Rows_query类型，可以看到对应的sql信息

  • row和statement总结

    1. 当写入的数据量较小时，row和statement所占用的空间差不多
    2. 当写入的数据量较大的时候(比如导入数据，或者批量操作时)，row要记录每行的变化，所以比较占用空间
    3. 且写入数据量很大时，row格式下，commit会比较耗时间，因为他还要写binlog(binlog只有在提交时才会写入)
    4. 假设更新一张几百万的表，产生的binlog可能有几百兆，当commit时，写入的数据量就是几百兆，所以会有阻塞等待的效果，但其实是在写binlog到磁盘而已

  • binlog_cache

    binlog默认写入到binlog_cache中

    默认32k，线程级别的变量，不建议设置太大

    当有一个大的事务时(几百兆)，内存中显然是放不下这么大binlog的，所以会记录到磁盘

    Binlog_cache_disk_use: 记录了使用临时文件写二进制日志到次数(做监控需要关注这个)

    Binlog_cache_use：记录了使用缓冲写二进制日志的次数

    写日志本来就挺慢的，现在cache中写不下，再写入到磁盘，然后再写binlog，就是两次写磁盘，就更慢了

    如果参数Binlog_stmt_cache_disk_use次数很多，就要看下binlog_cache_size设置是否太小了，或者事务本身太大了

    mysql在oltp的场景下，应该是很快的小事务，如果有大事务，应该把大事务拆成小事务去执行

  • binlog与redo的一致性

    使用内部分布式事务来保证一致性

    在commit时，(无论是用户自己输入，还是系统自动添加)，会有如下几个步骤

    1. innodb层写prepaere log
      • 写的还是redo log(或者就是redo log，只是内容不一样，这里不是记录页的变化)
      • 写入的是xid(事务id，准确的说，xid是写在undo页上的)
    2. mysql层写binlog
    3. innodb层写commit log
      • 同样写的是redo log file

    注意：这里说的写入指的是写入到磁盘(落盘成功)

    • 假设，如果没有第一步的prepare log，而是直接写第二步的mysql binlog，以及接着写第三步innodb commit log

      此时假设出现binlog写入成功，而innodb commit log写入失败的情况(比如宕机)，那随后机器重启的恢复时，就会对该事务回滚；

      如果此时binlog以及传递到了slave机器上，且slave上commit了，会导致主从数据不一致(master上回滚了)

    • 现在有了prepare log了以后，prepare log写入成功，假设还是binlog写入成功，而commit log写入失败的情况下

      此时事务恢复到时候，检查到prepare log写入成功，binlog写入成功，那就直接commit，而不管commit log是否写入成功

    • 如果prepare log写入成功，binlog写入失败，那么恢复时，也会回滚

    • 如果没开binlog，就没有第一步和第二步，只写第三步commit log，恢复到时候没有commit log，就会直接回滚

    总结：

    • 一个事务在prepare log中写入成功，在binlog中写入成功，那就是必须要提交(commit)
    • 用户/系统commit --> redo file(prepare log) --> binlog --> redo file(commit log)
    • xid即写入prepare log中，也会写入到binlog中，当恢复时，会对比一下某个xid在两个文件中是否都存在，如果都存在，该xid对应的事务才会提交
    • mysql5.6以后，写入binlog和写入commit log都是通过组提交的方式刷入到fsync到磁盘上的
    • mysql5.7以后，写入prepare log也是通过组提交的方式刷入(fsync)到磁盘上的(在写binlog之前执行一次fsync，就批量刷入prepare log)
    • 组提交失败了，并不会回滚改组中的所有事务，而是哪个失效了，就回滚哪个。

mysqlbinlog

增加了-vv参数可以看到注释信息，但是这些信息不是sql，它只是记录了页的变化

redo和binlog的总结

  • redolog时innodb层的，binlog时mysql层的
  • redolog时物理逻辑日志，binlog是逻辑日志
  • 写入的时间点一样
    • redo log可以多个写入点，比如master thread 刷新，log buffer使用量大于1/2，还有事务提交
    • binlog只有在事务提交的时候写入

undo

checkpoint

mysql的checkpoint是记录在redo log file(重做日志)的前2k字节中，(一个log block是512byte，所以是前四个log block)

存在两个checkpoint(CP1和CP2)，两个checkpoint循环写入

  • 所以说，redo log的写入其实不是完全顺序，因为还需要更新checkpoint中的内容
  • 通过判断两个checkpoint中的值大小，可以判断哪个checkpoint是最后写入的，即使使用小的checkpoint，也无非多恢复一些数据，多花一点时间而已
  • 设计层两个，主要是为了容灾考虑

undo对象

  • rollback segment 回滚段
  • undo log segment undo日志段
  • undo page
  • undo log
  • undo log record
rollback segment(回滚段)
  • mysql5.5前只有1个rollback segment
  • mysql5.5+有128个rollback segment
  • 不保存任何undo log
  • 仅保存undo log segment的位置
  • 含有1024个undo slot
  • mysql5.5中只有一个rollback segment，即只有1024个undo log segment，那就表示最多只能有1024个并发事务去执行undo
  • 如果用不到undo，其实是可以超过1024个线程的
  • 在mysql5.6中支持128*1024个并发执行undo的线程

undo默认存放在ibdata1中，即系统表空间中

系统表空间中的内容(ibdata1)

  1. double write
  2. 元数据信息(数据字典)
  3. insert buffer/change buffer
  4. undo信息(5.6后可以分离)，分开可避免竞争
    • innodb_undo_directory：表示undo存放的独立位置
    • innodb_undo_logs：表示rollback segment的个数
    • innodb_undo_tablespaces：存放到哪几个文件中去

在mysql5.7以后，rollback segment从128个减小等6个，剩余的32个rollback segment预留给临时表空间(ibtmp1)使用

undo log

  • undo log header
  • undo log records：undo log记录分为两种，insert的undo和update的undo
    • insert undo log record：记录insert
    • update undo log record：记录update和delete

undo是逻辑记录，记录了每一行修改的值(前后项)

之前说到redo写prepare日志，其实就是把上面中TRX_UNDO_STATE修改为PREPARE状态。(redo记录的是页的变化，undo也是页)

purge

  • 真正的删除记录
  • 删除undo log

举例：表tb1中有记录pk=1,2,3; 此时delete from tb1 where pk = 1;

  1. 将pk=1的记录标记删除(delete-mark, info bits), 数据库中pk=1的记录此时还是存在的，空间并没有释放，是同步操作(sql执行完了，也就标记完了)
  2. purge，该部分为后台线程(purge线程)异步操作，会真正的删除该记录，且空间被释放，purge线程是系统自动的，无法人为控制。

标记为删除的原因：

  1. 该事务可能需要回滚，先做保留
  2. 当事务1去删除pk=1且没有提交时，事务2应该要能看到pk=1的记录(事务的隔离性)

过滤条件时聚集索引

  1. delete：将该记录标记为delete-mark
  2. update：将该记录先物理delete(聚集索引主键相同的行最多只有1个)，然后insert(或者可以原地更新in place update)，即使删除了，也可以通过undo进行还原

过滤条件时二级索引

  1. delete：将该记录标记为delete-mark
  2. update：将该记录标记为delete-mark(索引列时columns + pk，即使时唯一索引更新也是和原来的不一样)，然后insert

为什么没有insert

  1. insert操作是不需要异步purge，因为insert的记录都是之前不存在的
  2. 不存在记录(为提交)是没有别的事务能引用到的，所以insert以后，对于的undo可以直接删除，而不需要等待异步purge

总结：

  1. delete-mark的记录最后会被purge线程回收，purge会检测记录上是否有其他事务在引用undo，如果没有就可以删除
  2. innodb_purge_threads(5.6版本以后)，可以设置的大一些，回收的速度快一些

分布式事务

xa start xid; 开启一个分布式事务

xa end xid; 结束一个分布式事务

xa prepare xid; 将分布式事务变成prepare状态

xa revover; 查看分布式事务

  1. 分布式事务是串行执行的(不能快照读)，在分布式事务中，使用两阶段事务，如果prepare成功了，就一定要提交
  2. 如果发生commit失败，事务就会变成悬挂事务，需要人工介入，查看prepare是否成功，而后决定commit或者rollback
