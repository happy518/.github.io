join算法

simple nested loop join

Simple nested loop join(简单嵌套循环链接)：可以理解成两个for循环，外循环走一次，内循环走n次(n等于外循环的次数)

  • 其算法伪代码如下:

For each row r in R do # 扫􏰀R表 Foreach row s in S do # 扫􏰀S表

If r and s satisfy the join condition # 如果r和s满足join条件 Then output the tuple # 那就输出结果集

  1. R表扫描了1次
  2. S表扫描了count(R)次
  3. 该方法相当于笛卡尔积，实际上数据库不会使用该算法

index nested loop join

index nested loop join(索引嵌套循环链接)：外表扫描一次，内表不会直接扫描，而是查找内表相应的索引的方式，和外表的记录进行匹配

  • 算法伪代码如下：

    • 具体的执行过程：

      1. 从表R中读入一行数据r
      2. 从数据行r中，取出join字段到表S的索引上查询
      3. 取出S中满足条件的行，根r组成1行，作为结果集的一部分
      4. 重复执行1到3，直到R表的末尾，循环结束

    • 优化思路

      主要是为了减少内层表数据的匹配次数，简单来说index nested loop join就是通过外层表匹配条件，直接与内层表索引进行匹配，避免了和内层表的每条记录进行比较，极大了减少了对内层表的匹配次数，从原来的匹配次数=外层表行数 * 内层表行数，变成了外层表行数 * 内层表索引的高度，极大的提升了join的性能。

    1. S表上有索引，R表上可以没有索引
    2. 扫描R表，将R表的记录和S表的索引进行匹配
    3. 优化器倾向使用记录数少的表作为外表(又称驱动表)
    4. 如果数据量大，index nested loop join的成本也是很高的，尤其是在二级索引的情况下，需要大量的回表操作

    注意：

    使用index nested loop join算法的前提是匹配的字段必须建立了索引

block nested loop join

block nested loop join(缓存块嵌套循环链接)：将外表中需要join匹配的列(不是完整的记录)，暂时保存到一块内存(join buffer)中，让后将这块内存中的数据和内表进行匹配，内表只扫描一次，可用于链接是all，index，range的类型

  • 其算法伪代码：

  • 具体的执行过程：
    1. 把表R的数据读入线程内存join_buffer中
    2. 扫描S表，把S表的每一行取出来，根join_buffer中的数据进行比较，满足join的，作为结果集的一部分返回
    3. 如果R表的数据太大，join_buffer一次放不下，会分段处理，一块一块的加载到内存
    4. 加载R表的一部分数据到join_buffer，执行1，2后，在清空join_buffer，在继续扫描R表，进行1，2操作

block nested loop join与simple nested loop join相比，多了一个join buffer

128m, 默认256k

join buffer用的不是buffer pool中的内存，而是线程级别的内存，可以通过explian查看执行计划，通过join字段的大小，来预估join_buffer_size的大小

注意：

增大join_buffer_size优化的前提是没有使用index，如果使用了index，根本不会使用block nested join算法

join算法总结

不论是index nested loop join还是block nested loop join都是在simple nested loop join的算法基础上进行优化的。

index nested loop join是通过索引的机制减少内层表的循环匹配次数来达到优化效果的。

个人理解：

eg： select * from tbl_join_test_1 as a, tbl_join_test_2 as b where a.a = b.a; # b表的a列添加了索引

index join匹配过程，扫描外表a中的每一条记录r的a列，去内表b中查询，而simple join的匹配是扫描外表a中的每一条记录r，再去扫描内表b的每一条记录i，比较r的a列和i的a列是否相等

所以说，index是优化了内层表的循环次数

block nested loop join是通过一次缓存多条数据批量匹配的方式来减少外层表的IO次数，同时也减少了内层表的扫描次数。

  1. 永远用小结果集驱动大结果集。(其本质就是检查外层表的数据量)
  2. 为匹配字段增加索引(减少内层表的循环匹配次数)
  3. 增大join_buffer_size大小，一次性缓存的数据越多，那么内存表扫描次数就越少
  4. 减少不必要的字段查询，字段越少，join buffer所缓存的数据就越多
